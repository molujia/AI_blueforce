## 一、严重问题（High Priority）

### 1.1 路径规划评分公式存在缺陷

**文件：** [zhibing/routing/path_planner.py](zhibing/routing/path_planner.py#L32)

**问题：** `total_score = distance + risk + time_score` 中，`distance`（米，数量级 100-1000）和 `time_score`（distance/5.0，同样大）直接与 `risk`（0-200 的小浮点数）相加，导致风险分数在总评分中微不足道，无法有效影响路径选择。

**建议修改：**
```python
# 修改前
total_score = distance + risk + time_score

# 修改后：归一化到可比尺度
MAX_DISTANCE = 2000.0  # 场景最大距离
MAX_RISK = 200.0
distance_norm = distance / MAX_DISTANCE * 100
risk_norm = risk / MAX_RISK * 100
time_norm = time_score / (MAX_DISTANCE / 5.0) * 100
total_score = distance_norm + risk_norm + time_norm
```

### 1.2 Socket 传输层 TCP 消息截断风险

**文件：** [zhibing/adapter/socket_transport.py](zhibing/adapter/socket_transport.py#L30-L34)

**问题：** `conn.recv(1024 * 1024)` 单次读取固定 1MB，但 TCP 是流协议，单次 `recv` 可能返回不完整的 JSON。发送端用 `\n` 分隔（第 30 行），但接收端没有循环读取直到 `\n`。导致大于 1MB 的响应被截断，或分段到达的消息解析失败。

**建议修改：**
```python
# 增加基于分隔符的帧协议
data = b""
while True:
    chunk = conn.recv(4096)
    if not chunk:
        break
    data += chunk
    if b"\n" in data:
        break
response = json.loads(data.decode("utf-8"))
```

### 1.3 HTTP 传输层缺少错误处理

**文件：** [zhibing/adapter/http_transport.py](zhibing/adapter/http_transport.py#L34-L35)

**问题：** 对非 2xx 响应（400/401/403/500）没有处理，`urlopen` 会直接抛出 `HTTPError`，传播到调用方。没有重试逻辑、没有熔断器、没有有意义的错误消息。

**建议修改：** 增加 try/except 捕获 `HTTPError` 并转换为领域异常；增加指数退避重试（最多 3 次）。

### 1.4 坐标转换严重 Bug

**文件：** [zhibing/core/coord_service.py](zhibing/core/coord_service.py#L76)

**问题：** `to_wgs84` 方法中使用 `origin["lat"]` 代替 `clean["lat"]` 计算纬度，导致 VBS→WGS84 转换始终输出 `origin_lat + delta`，完全忽略输入坐标的实际纬度。

```python
# 错误（第 76 行）
lat = math.radians(float(origin["lat"])) + float(clean["y"]) / EARTH_RADIUS_M

lat = math.radians(float(clean["lat"])) + float(clean["y"]) / EARTH_RADIUS_M
```

### 1.5 Django 视图层模块级全局单例导致并发问题

**文件：** [zhibing/web/command_ui/views.py](zhibing/web/command_ui/views.py#L19-L21)

**问题：**
```python
SYSTEM = ZhibingDecisionSystem()
MEMORY = SessionMemory()
SCENARIO_ID = "demo_encirclement_v0"
```
模块级单例在 WSGI 多进程/多线程下共享状态。`SessionMemory` 使用 SQLite，并发请求会导致 `database is locked` 错误。且无法进行单元测试（无法 mock）。

- 或使用连接池、改用 Django ORM 管理会话
- 至少加 `threading.Lock` 保护 SQLite 写操作

### 1.6 Django settings.py 安全隐患

**文件：** [zhibing/web/zhibing_web/settings.py](zhibing/web/zhibing_web/settings.py#L6-L11)


### 1.7 状态管理器重复创建时静默覆盖

**文件：** [zhibing/core/task_state_machine.py](zhibing/core/task_state_machine.py#L15-L16)

**问题：** `StateManager.create()` 在 `task_id` 已存在时静默覆盖现有状态机，不抛出异常。这会导致状态丢失和难以追踪的 bug。

**建议修改：** 增加重复检查：
```python
def create(self, task_id: str, machine: TaskStateMachine) -> None:
    if task_id in self.machines:
        raise ValueError(f"Task {task_id} already exists")
    self.machines[task_id] = machine
```

### 1.8 分布式事务原子性问题

**文件：** [zhibing/main.py](zhibing/main.py#L85-L92)

**问题：** `submit_task` 在两个事务块之间调用。如果第一个事务提交成功、`submit_task` 到下层仿真也成功，但第二个事务（ACK/RUNNING 状态更新）失败回滚，系统已向下层提交任务但上层没有持久化记录。

**建议修改：** 将 `submit_task` 的调用放在事务内，或使用补偿事务（Saga 模式），或在第二个事务失败时向下层发送取消指令。

### 1.9 PDF 解析器对 CJK 文本无效

**文件：** [zhibing/knowledge/document_loader.py](zhibing/knowledge/document_loader.py#L100-L109)

**问题：** 
- `re.findall(r"\(([^()]{3,})\)", decoded)` 无法处理嵌套括号
- 现代 PDF 使用 FlateDecode 压缩流，此回退方法可能返回零文本
- 回退词正则 `[A-Za-z][A-Za-z0-9,.;:'\-/]{3,}` 只匹配拉丁字母，完全忽略中日韩文字

**建议修改：** 优先使用 `pypdf` 的高级 API（`PdfReader`）提取文本；增加 CJK 字符的正则支持；对压缩流进行解码。

### 1.10 前端渲染未检查 API 错误

**文件：** [zhibing/web/command_ui/static/command_ui/app.js](zhibing/web/command_ui/app.js#L54-L56)

**问题：** `loadDemoScene` 和 `submitRouteConstraint` 中 `renderProjection(data.projection)` 未检查 `data.projection` 是否存在。服务器返回 500 时，`response.json()` 仍会解析，但 `.projection` 为 `undefined`，导致后续函数崩溃。

**建议修改：**
```javascript
const data = await response.json();
if (!data.ok || !data.projection) {
    renderChatMessage("system", "场景加载失败：" + (data.error || "未知错误"), true);
    return;
}
renderProjection(data.projection);
```

### 1.11 LLM 调用静默吞掉所有异常

**文件：** [zhibing/knowledge/llm_router.py](zhibing/knowledge/llm_router.py#L37-L49)

**问题：** `extract_json` 的 except 块捕获所有异常（网络错误、认证失败、JSON 解析错误、模型超时）并静默返回空结果 `{"entities": [], "relations": [], "rules": [], "fallback": True}`。调用方无法区分"无实体"和"LLM 调用失败"。

### 2.9 `ConstraintPatch` 缺少数值验证

**文件：** [zhibing/routing/constraint_patch.py](zhibing/routing/constraint_patch.py#L10-L18)

```python
    if self.action not in ("avoid", "prefer", "priority", "ignore_zone"):
        raise ValueError(f"Invalid action: {self.action}")
```

**文件：** [zhibing/adapter/sqf_compiler.py](zhibing/adapter/sqf_compiler.py#L59-L81)

**问题：** `_sqf_value` 递归编码嵌套结构，无环检测。若传入自引用 dict（如 LLM 生成的数据），会导致栈溢出。

**建议修改：** 增加 `seen_ids` 集合进行循环引用检测。

### 3.11 `llm_migration_client` 重试过于宽泛

**文件：** [llm_migration_client.py](zhibing/llm_migration_client.py#L162-L170)

**问题：** 重试循环 `except Exception as exc` 捕获所有异常，包括编程错误（如 `TypeError`）也会重试。

**建议修改：** 仅重试 `httpx.NetworkError`、`httpx.TimeoutException` 和 5xx HTTP 状态码。

### 3.12 `sqf_compiler` 的 `chr(91)` / `chr(93)` 混淆

**文件：** [zhibing/adapter/sqf_compiler.py](zhibing/adapter/sqf_compiler.py#L62-L63)

**问题：** 使用 `chr(91)` 代替 `[`、`chr(93)` 代替 `]`，无注释说明原因，损害可读性。

**建议修改：** 直接使用 `[` `]` 字符，或添加注释解释为何这样做。

## 四、架构与设计建议

### 4.1 统一错误处理体系

当前系统缺乏统一的异常层次结构。建议：

```
ZhibingError (基类)
├── IntentRecognitionError
├── SceneGatheringError
├── PlanningError
├── BTSelectionError
├── ParameterGenerationError
├── TransportError
│   ├── HttpTransportError
│   └── SocketTransportError
├── StateManagementError
└── KnowledgeQueryError
```

### 4.2 引入结构化日志

当前系统无任何日志记录。建议在关键路径增加 `logging`：
- LLM 调用成功/失败及耗时
- 路径规划决策及依据
- 下层仿真通信状态
- 异常吞掉的 catch 块（至少 WARNING 级别）

### 4.3 配置管理统一化

当前 `config.py`、`llm_migration_client.py`、`llm_router.py` 各有自己的配置读取逻辑和重复默认值。建议：
- 所有可配置值统一从 `config.py` 导出
- 使用 `dataclass` + 环境变量加载 (如 `pydantic-settings`)
- 消除硬编码在各文件中的重复默认值

### 4.4 前端状态管理

当前 `app.js` 使用全局变量管理状态，随着功能增加会变得难以维护。建议引入轻量状态管理（如简单的 pub/sub 模式），将地图状态、路线状态、会话状态、UI 状态分离管理。

### 4.5 知识库文档热加载

当前知识库在启动时构建后永久缓存。建议：
- 基于文件 mtime 的自动失效
- 提供 `/api/knowledge/reload` 管理端点
- 使用 `watchdog` 监控 `test_files` 目录变化

### 4.6 前端增加 Content Security Policy

当前 `index.html` 无 CSP 头。建议至少设置：
```html
<meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'">
```
